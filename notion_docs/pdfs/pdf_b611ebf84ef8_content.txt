#### ページ1: エグゼクティブサマリー

**【エグゼクティブサマリー】CS業務改善・自動化推進プロジェクト**

##### 今回の要点 (What happened)
*   CS業務を「サービス改善(数値目標)」と「業務改善(ルーティン効率化)」の2軸に再定義。
*   現状のルーティンタスクを棚卸しした結果、約280個の未整理タスクを特定。
*   CTO室との定例連携(毎週金曜)にて、順次自動化の判断と実装を進める体制へ移行。
    *   **図表の説明:** 右側に上向きと左右に分かれる三方向の矢印のアイコンがあり、業務の再定義と方向転換を示唆しています。

##### 統括に関係する論点 (So What)
*   自動化推進のためには、属人化している280業務の「ワークフロー可視化」が最優先課題。
*   システム移行期において、顧客データ重複防止のため一部「目視確認」の工程維持が必要。
*   月次資料作成(10時間/月)など、ROIの高い業務からリソースを再配分する必要がある。
    *   **図表の説明:** 右側に三角の中に感嘆符が描かれた警告アイコンがあり、重要な論点や課題を強調しています。

##### 本日求めたい判断・指示 (Now What)
*   [ ] 全メンバーによる「業務フロー図作成(AI活用)」工数投下の承認。
*   [ ] 顧客アカウント発行プロセスにおける「目視チェック工程」残存の承認。
*   [ ] 数値管理シート(KPI管理)自動化への担当者(大河)アサインの承認。

#### ページ2: 【情報共有】CS業務の「2軸化」によるリソース配分方針の再定義

##### 課題・背景
*   **現状の課題:** 「数値を追う業務」と「日々発生する作業」が混在し、優先順位が曖昧化。
*   **方針変更:** 以下の2軸で業務・ミーティングを分離して管理する。
    *   **CS Operations** (図で階層的に示されている)
        *   **サービス改善 (Value)**
            *   売上・数値目標に直結 / 効果検証必須
        *   **業務改善 (Efficiency)**
            *   ルーティン作業 / CTO室主導で自動化

##### 対応・判断事項
*   **判断ポイント:** 特になし(方針共有のみ)。
*   **決定事項:**
    *   今後のCTO室定例MTGは「業務改善」に特化して議論する。
    *   施策実施時は「どちらの軸か」を明確にし、サービス改善施策であればROIの事前定義を必須とする。

#### ページ3: 【決定】約280個のルーティンタスク可視化と全メンバーによるワークフロー作成

##### 課題・背景
*   **280 Tasks** (大きく表示された数値)
*   業務棚卸しの結果、約280個のタスクをリストアップ済み。
*   **課題:** 「何が自動化可能か」を判断するためには、各タスクのフロー図(手順)が必要だが、現状は担当者の頭の中にしかない(属人化)。
*   **手法:** 音声入力+ChatGPT活用により、工数をかけずにフロー図を作成する体制を準備済み。
    *   **図表の説明:** 音声入力アイコン → AI処理アイコン → ワークフローチャートアイコン という流れで、AIを活用したワークフロー作成プロセスを示しています。

##### 対応・判断事項
*   **判断ポイント:** 全メンバーのリソースを一時的に「ワークフロー作成」へ割くことを承認するか。
*   **決定事項:**
    *   全CSメンバーは、担当業務のワークフロー図を作成する。
    *   入学式対応やオフ会準備など、ボリュームの大きいタスクから優先的に着手。
    *   作成したフロー図をもとに、CTO室が自動化可否を判定する。

#### ページ4: 【確認】システム移行に伴う顧客データ重複防止策(目視チェックの維持)

##### 課題・背景
*   **システム制約:** 顧客情報(氏名・クラス名)の入力において、表記ゆれ(例: shi / ci)による重複登録リスクがある。
*   **リスク:** 完全自動化すると、同一人物の二重アカウント発行等の事故につながる可能性がある。
    *   **図表の説明 (問題点):** Form Input → Auto-Check → Fail due to 'shi/ci' → Duplicate Account Created (Risk) のフローが示されており、自動チェックのみでは表記ゆれによる重複登録リスクがあることを赤いXマークで示しています。

##### 対応・判断事項
*   **判断ポイント:** リスク回避のため、一部手動工程が残ることを許容するか。
*   **決定事項:**
    *   アカウント発行プロセスにおいて、最終的な「目視チェック工程」を維持する。
    *   完全自動化よりもデータの正確性を優先し、ひろし(開発担当)にてシステム側の突合ロジック実装と並行して運用する。
    *   **図表の説明 (解決策):** Form Input → System Check → Manual Visual Check → Account Created (Safe) のフローが示されており、システムチェックに加えて目視チェックを挟むことで安全にアカウントが作成されることを緑のチェックマークで示しています。

#### ページ5: 【決定】月次数値管理シート(KPI資料)の完全自動化

##### 課題・背景
*   **現状:** 月次報告資料のデータ集計を、担当者(トナリ)が毎月10時間以上かけて手作業で実施中。
    *   **図表の説明:** 時計のアイコンと「10 Hours/Month Manual Work」の矢印が示され、手作業にかかる時間コストを表しています。
*   **リスク:** 手動集計によるヒューマンエラーが発生しやすく、正確な数値判断の阻害要因となっている。
*   **リソース:** 大河(Taiga)が即時対応可能。

##### 対応・判断事項
*   **判断ポイント:** 大河の工数を本件の自動化開発に最優先で割り当てるか。
*   **決定事項:**
    *   大河が数値管理シートの自動化実装を担当する。
    *   単なる自動化だけでなく、データの定義や分析マニュアルの整備もセットで行い、属人性を排除する。
