各部署のデータ活用推進のため、会社全体に適用できるデータ基盤の整理を進行中。
withAIドリルを総会でローンチ
1. AI活用度レベル定義の策定
これまでAI活用は人によって粒度がバラバラで、「できている／できていない」を共通言語で語れなかった。
評価・育成・配置のいずれも、共通スケールがないままでは設計が難しい。
習慣 → スキル → 成果物 → 影響範囲の順で並べた 5段階のレベル定義 を整備。
各レベルは10項目のチェックリストで構成、8項目以上の達成で次レベルへ昇格。
運用方針
3ヶ月ごとにレベルを1段階引き上げる想定。
中期ゴール（4期目終了時）
  今いる全員が Lv.3 以上（＝個人最適ではなく、チームを動かせるレベル感）に到達。
CTO室メンバーの入れ替え

4. データ基盤の整理
目的
「各部署でデータドリブンに動ける状態」を実現するための土台整備。
AI を業務に組み込むには、まずコンテキストとなるデータが部署横断で参照可能 であることが前提。
進行中の作業
各部署のデータの所在・形式・更新頻度の棚卸し。
部署ごとに散在しているスプレッドシート／個別DBの集約方針整理。
共通のデータ参照導線（BI／ダッシュボード）の設計準備。
これらが完成後、Levelaの情報を参照できるSkillsを作成しgithubで配布
5. AIドリルの構築（全社展開予定）
5-1. 目的
レベル定義（Lv.1〜Lv.5）を実務スキルとして定着させるための学習基盤。
完成後は全社展開し、各部署ごとに対戦形式でAI活用力を競わせる運用に移行する。
5-2. 構成
第1章: AI初級編 — Lv.1相当の基礎（プロンプト基礎、ツール使い分け、機密情報の扱い 等）
第2章以降: 既存の学習シリーズ群を順次組み込み
  Git完全マスターシリーズ（8コース）
  Web開発基礎シリーズ（8コース）
  Web開発実践シリーズ（8コース）
  UIデザイン基礎シリーズ（7コース）
  UIデザイン実践シリーズ（7コース）
5-3. 展開イメージ
完成 → 全社員に配布 → 部署対抗の進捗・到達度で競争。
部署ごとのレベル分布が可視化されるため、CTO室の育成ターゲット選定とも連動。
GitHub Organization 移行 + PR ベースの決裁フロー
会社アカウントへの一本化
これまで個人アカウント混在で運用されていた状況を整理。
Skills／作成プロジェクトは、すべて levela-inc Org 内で完結させる。
  Skillsが格納されたら図解付きでDiscordに通知が飛ぶように
資産が個人ではなく会社に蓄積される／権限管理を明確化／退職・異動時の引き継ぎが容易になる。
PR ベースの決裁フロー（5/11 目安）
幹部陣の決裁プロセスをプルリクエスト形式に統一。
申請者は変更内容（仕様・コスト・期日 等）を PR にまとめ、レビュアーに送付。
レビュアーは 役員陣または代表。決裁内容に応じて必要な役割（例：技術／コスト／法務）を Required Reviewer として割り当て。
マージ ＝ 決裁承認。誰がいつ何を承認したかの履歴が Git に残る。
6-3. 期待される効果
決裁の属人性を排除（承認ルートと役割が明示される）。
決裁履歴が自然に蓄積され、後追い・監査が容易。
決裁スピードの可視化（PR ごとのリードタイム計測）。
7. GitHub アクティビティの可視化
7-1. 目的
誰が今、何をしているか を、Slack の報告や口頭ではなく コミット履歴ベース で常時把握できるようにする。
役員・上長は、各メンバーの稼働状況／着手中タスク／直近の成果物を即時に確認できる。
7-2. 仕組み
メンバーごとの活動ダッシュボードを構築（自動更新）。
表示項目：直近のコミット内容、稼働中リポジトリ、本日のコミット数、PR の open / merge 状況。
段階展開：個人ダッシュボード → 部署ダッシュボード → 全社ダッシュボード。
7-3. コミット粒度の標準化
コミットは hook で自動管理 することで、人によるコミット頻度の差を発生させない。
全メンバーを同じスケールで比較可能に。
「コミットしていない＝動いていない」と誤って判断される現象を防ぐ。
7-4. 期待される効果
進捗確認コストの削減（聞かなくても見れば分かる）。
着手停滞・ボトルネックの早期発見。
評価・育成判断に使える客観データの自然蓄積。
