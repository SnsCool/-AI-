🎯 Webの全体像（このシリーズの要約）
① そもそもWebって何？
スマホで検索する、SNSを見る、動画を見る——これらに共通するのは「別の場所にある情報をお願いして受け取っている」こと。スマホの中にぜんぶ入っているわけではなく、毎回どこかの「お店」にお願いして、料理（データ）を運んできてもらっている。このお願いと配達の仕組みが Web。
インターネット = 世界中のコンピュータをつなぐ「道路網」
Web = その道路の上を走る「配送サービスの一つ」（メールや動画通話も別の配送サービス）
Webサイトが見えない時、まず「インターネット自体は動いてる？（メールは届く？）」を確認すると、配線の問題かサービスの問題かを切り分けられる。
② レストランの例で理解するブラウザとサーバー
ブラウザは料理（データ）をそのまま画面に映しているわけではない。届いたデータを「読み解いて」「文字の大きさ・色・配置を計算して」「画面に組み立てる」——つまり設計図を読んで家を建てる大工のような存在。Chrome と Safari で同じサイトの見た目が微妙に違うのは、大工（ブラウザ）が違うから。
③ Webページの中身は「3つの担当」に分かれている
サーバーから届く「設計図」は、3つの別の言語で書かれている。それぞれ役割がはっきり分かれているので、修正もしやすい。
3つに分けるメリット：
担当が違えば担当者も別にできる（HTMLの人/CSSの人/JSの人）
着せ替えができる（HTMLそのままでCSSだけ変えるとデザイン刷新）
問題箇所が特定しやすい（見た目が変なら CSS を疑う、動かないなら JS を疑う）
④ HTML：意味のある骨組みを作る
HTMLは「見た目を作る言語」ではなく「意味を伝える言語」。<h1> は「大見出し」、<p> は「段落」、<a> は「リンク」と、それぞれのタグが「これは何か」を伝える。
タグ：<タグ名>中身</タグ名> の形で囲む（開始タグと終了タグ）
属性：<a href="..."> のようにタグに追加情報をつける（住所、画像のパス、IDなど）
ネスト：タグの中にタグを入れる入れ子構造。親子関係でページの構造ができる
セマンティック：<div> ばかりでなく <header> <nav> <article> <footer> を使うと、検索エンジンも読み上げソフトも理解できる
⑤ CSS：誰に・何を・どう変えるか
CSSは「誰に（セレクタ）」「何を（プロパティ）」「どう変えるか（値）」の3つで指示する。
セレクタ {
  プロパティ: 値;
}
⑥ JavaScript：ページに動きと反応を加える
HTMLとCSSだけだと、ページは「読むだけの紙」。JavaScriptを加えると「触ると反応する家」になる。
DOM：ブラウザが組み立てた「ページの構造ツリー」。JSはこれを操作してページを書き換える
イベント：「クリックされた」「キーが押された」「フォームが送信された」などのできごとを感知する仕組み
イベントハンドラ：イベントが起きたときに実行する処理（"〜が起きたら〜する"）
静的ページ vs 動的ページ：JSなしは静的（変わらない）、JSありは動的（書き換わる）
JavaScript はブラウザの中だけではなくサーバー側（Node.js）やスマホアプリ（React Native）でも動く、汎用性の高い言語。
⑦ HTML / CSS / JavaScript が揃って初めて「フロントエンド」
この3つを使って「ブラウザの中で動く部分」を作ることをフロントエンド開発と呼ぶ。逆に、サーバー側（厨房側）で動くデータベース処理やビジネスロジックを作ることをバックエンド開発と呼ぶ（コース6で扱う）。

💡 学びのメタ原則
「日常の例え」で覚える
  ブラウザは大工、サーバーは厨房、HTMLは骨組み、CSSは内装、JSは家電。技術用語そのものより、役割と関係性を覚える方が応用が利く。
役割分担を意識する
  HTMLは「意味」、CSSは「見た目」、JSは「動き」。1つの技術で全部やろうとしないのが現代Web開発の鉄則。混ぜると修正が難しくなる。
トラブルは「切り分け」で解く
  「Webサイトが見られない」と言われたら、まず「メールは届く？」（インターネット自体は？）→「他のサイトは見られる？」（特定サーバーの問題？）→「他のブラウザでは？」（ブラウザの問題？）と範囲を狭めていくのがプロのデバッグ。
裏側を覗く習慣を持つ
  F12キーで開発者ツールが開く。ブラウザが受け取った設計図（HTML）、計算した見た目（CSS）、起きた通信（Network）、JSのエラー（Console）が全部見える。書ける前にまず読めるようになるのが上達の近道。
「同じデータでも見え方が違う理由」を説明できる
  ブラウザ違い・画面サイズ違い・OS違いで微妙に見た目が変わる。それはブラウザがデータを解釈して組み立てているから。「データが違うのか、解釈が違うのか」を区別できると、不具合の原因が早く見つかる。

  ゴール: Webの全体像、ブラウザとサーバーの役割、URL・HTTP・クライアント/サーバーの基礎を「日常の例え」で身につける。
  Lesson 1: Webの世界一周ツアー
  Q1
  問題: 検索する、SNSを見る、動画を見る。これらに共通していることは何でしょう？
  正解: A. どれも別の場所にある情報を受け取っている
  解説: 検索もSNSも動画も、すべて別の場所から情報を受け取っています。
  スマホの中に全データが入っているわけではなく、毎回どこかに「お願い」して情報をもらっているのです。
  Q2
  問題: ブラウザでURLを入力してEnterを押すと、ページが表示されます。
  この一瞬の裏側では、何が起きているでしょう？
  正解: A. ブラウザが別のコンピュータにデータを取りに行っている
  解説: ページはブラウザの中に最初から入っているわけではありません。Enterを押した瞬間、ブラウザはサーバーにデータを取りに行っています。
  この「見えないやり取り」が、一瞬の裏側で起きているのです。
  Q3（◯×）
  問題: Webページのデータは端末内に保存されており、ネット接続なしでも閲覧できる。
  正解: 間違い
  解説: Webページのデータは端末に保存されていません。ページを開くたびにサーバーから取り寄せているので、接続が切れると表示できなくなります。
  端末に一時的にデータが残ることもありますが、それは古いコピーであり、原本はサーバー側にあります。
  Q4
  問題: Webの仕組みをレストランに例えると、「あなたがブラウザでページを開く」のは、どの場面に似ているでしょう？
  正解: A. お客さんが料理を注文する場面
  解説: ブラウザでページを開くのは、レストランで料理を注文するのと同じです。
  あなた（ブラウザ）が「このページをください」とお願いすると、厨房（サーバー）が料理（データ）を準備して届けてくれます。
  Q5（マッチング）
  問題: 検索・SNS・動画視聴では、それぞれ「何を注文」して「何が届く」でしょう？
  「注文」と「届くもの」を対応させてください。
  正解:
  検索ワードを入力 ↔ テキストデータ
  SNSのタイムラインを開く ↔ 画像と文字のデータ
  動画のサムネイルをタップ ↔ 映像と音のデータ
  解説: どの操作も「お願い（注文）→ データ（料理）」という同じパターンで動いています。
  届くデータの種類は違っても、裏側では毎回サーバーに「これをください」とお願いし、データが返ってきているのです。
  Q6（穴埋め）
  問題: ブラウザを使ってインターネット上で情報を公開・閲覧できる仕組みには、専用の名前があります。
  正解: Web
  解説: この仕組みを「Web」と呼びます。Webはブラウザを通じて情報を公開・閲覧するための仕組みです。
  Wi-Fiは通信の手段であり、情報を公開・閲覧する仕組みそのものではありません。BluetoothやGPSもWebとは別の技術です。
  Q7
  問題: Webでは「ブラウザ」と「サーバー」がデータをやり取りしています。
  データを保管し、ブラウザからのお願いに返答しているのはどれでしょう？
  正解: A. サーバー
  解説: データを保管して返答しているのはサーバーです。ブラウザはお願いを送る側、インターネットはお客さんと厨房をつなぐ通路のような存在です。
  ブラウザからサーバーに「お願い」が飛び、サーバーからブラウザに「返答」が戻る流れになっています。
  Q8
  問題: 「ブラウザ」は、実際には何をしているアプリでしょう？
  正解: A. サーバーと会話し、データを画面に組み立てている
  解説: ブラウザはただ表示するだけではなく、サーバーと会話し、受け取ったデータを画面に組み立てています。
  サーバーから届くのは「完成した画面」ではなく「材料（データ）」です。ブラウザがその材料を使って、あなたが見る画面を仕上げているのです。
  Q9（◯×）
  問題: Wi-Fiには繋がっているのに、あるWebサイトだけ表示されない。
  この場合、原因はインターネット接続ではなくサーバー側にある可能性が高い。
  正解: 正しい
  解説: Wi-Fiに繋がっているならインターネット接続は正常です。特定のサイトだけ表示されないなら、そのサイトのサーバーに問題がある可能性が高いと判断できます。
  ブラウザ・インターネット・サーバーの役割を知っていると、このように原因を絞り込めます。
  Q10
  問題: ここまでの内容を振り返ります。
  Webの仕組みを最も的確にまとめているのはどれでしょう？
  正解: A. ブラウザがサーバーからデータを取り寄せ、画面に仕上げる仕組み
  解説: Webとは、ブラウザがサーバーにデータを取りに行き、届いた材料から画面を組み立てる仕組みです。
  普段何気なく使っている検索やSNSも、この「お願い→返答→組み立て」の繰り返しで動いています。
  Lesson 2: インターネットとWebの違い
  Q1
  問題: 「ネット」と「Web」——この2つは同じ意味でしょうか？
  正解: A. 違う意味。「ネット」と「Web」は別のもの
  解説: 実はインターネット（ネット）とWebは別物です。インターネットは世界中を繋ぐ通信網、Webはその上で情報を公開・閲覧する仕組みです。
  この違いが分かると、Webの仕組みがグッと見えてきます。
  Q2
  問題: 自宅のインターネット回線が止まりました。パソコンで使えなくなるのはどれ？
  正解: A. Webサイト閲覧、メール、ビデオ通話のすべて
  解説: インターネット回線が止まると、すべてのオンラインサービスが使えなくなります。Webもメールもビデオ通話も、すべてインターネットの上で動いているからです。
  インターネットは「全サービス共通の土台」なのです。
  Q3
  問題: インターネットを「道路網」に例えると、Webはどれに当たるでしょう？
  正解: A. 道路の上を走る「配送サービス」のひとつ
  解説: インターネットが「道路網」なら、Webはその上を走る「配送サービス」のひとつです。
  タクシーは「人を運ぶ」サービスですが、Webは「情報を届ける」ので配送が最も近い例えです。道路の上にはメールやビデオ通話など他のサービスも走っています。
  Q4
  問題: 「Web」という仕組みが関わっているのは、主にどんな場面？
  正解: A. ブラウザでサイトやページを見る
  解説: Webはブラウザでサイトやページを閲覧する仕組みです。メールや電話はWebとは別の仕組みで動いています。
  すべてインターネットという同じ道路を使いますが、走る「車」が違うのです。
  Q5（穴埋め）
  問題: 世界中のコンピュータやスマホが繋がっている巨大なネットワークには、専用の名前があります。
  正解: インターネット
  解説: インターネットは、世界中のコンピュータを繋ぐ「ネットワークのネットワーク」です。
  Webはその上で動くサービス、サーバーは情報を保管する場所。それぞれ役割が違います。
  Q6（マッチング）
  問題: 「道路とサービス」の比喩と、実際の技術用語を正しく対応させてください。
  正解:
  道路網 ↔ インターネット
  配送サービス ↔ Web
  メールサービス ↔ メール
  解説: インターネット＝道路網、Web＝配送サービスです。道路は1つですが、その上を走るサービスは何種類もあります。
  Q7（◯×）
  問題: Webサイトが見られなくても、メールが届くなら、インターネット自体は正常に動いている。
  正解: 正しい
  解説: Webとメールはインターネット上の別々のサービスです。メールが届くということは、道路（インターネット）は通行可能で、Webサービス側に問題があると判断できます。
  この「切り分け」がトラブル解決の第一歩です。
  Q8
  問題: 4つのサービスがインターネットの上で動いています。
  この中で、Webとは別の仕組みとして生まれたものはいくつある？
  正解: A. 3つ（メール、ビデオ通話、ファイル転送）
  解説: メール、ビデオ通話、ファイル転送はそれぞれWebとは別の仕組みとして生まれました。今はブラウザ上で使えるものもありますが、もとの仕組みはWebとは別物です。
  インターネットという同じ道路の上を、それぞれ違う「車」で走っているイメージです。
  Q9
  問題: 友達が「Webサイトが見られない。インターネットが壊れた？」と言っています。
  あなたならどう答えますか？
  正解: A. 他のアプリやメールも使えるか確認してみよう
  解説: インターネットとWebは別物なので、他のサービスが使えるかで原因を切り分けられます。
  ビデオ通話やメッセージアプリが使えるなら、インターネットは生きていてWebサービス側の問題。全部使えないなら接続の問題です。
  Q10
  問題: インターネットとWebの違いを知ると、できるようになることは何でしょう？
  正解: A. トラブル時に「接続の問題か、サービスの問題か」を判断できる
  解説: この区別が分かると、トラブルの原因を切り分ける力が手に入ります。「全部使えない→接続の問題」「Webだけ使えない→サービスの問題」と判断できます。
  友達に「ネットとWebって実は別物なんだよ」と説明できるようになりました。
  Lesson 3: ブラウザの正体
  Q1
  問題: Chrome と Safari で同じWebサイトを開いたのに、表示が微妙に揃わない。
  なぜ？
  正解: A. ブラウザごとにデータの解釈の仕方が異なるから
  解説: サーバーが返すのは完成した画面ではなく、設計図のようなデータです。
  そのデータの解釈の仕方がブラウザごとに微妙に異なるため、文字の大きさや余白の取り方などで仕上がりに差が出ます。
  Q2（◯×）
  問題: ブラウザは、サーバーから届いた完成済みの画面をそのまま映しているだけである。
  正解: 間違い
  解説: ブラウザは届いたデータを自分で組み立てて画面にしています。
  「そのまま映す」のではなく、文字の大きさや色を決めたり、画像を配置したりといった処理を裏側でこなしています。
  Q3
  問題: ブラウザの働きを身近なものに例えると、どれが一番近い？
  正解: A. 設計図を読んで家を建てる大工
  解説: ブラウザは「設計図を読む大工」です。サーバーから届くのは「設計図（データ）」であり、それを読み解いて「家（画面）」に組み立てるのがブラウザの仕事。
  映写機のように「そのまま映す」のではなく、データを解釈して形にする点がポイントです。
  Q4
  問題: 同じ設計図（データ）なのに Chrome と Safari で見た目が微妙に違う。
  「大工」の例え話で考えると、なぜ？
  正解: A. 大工が違えば、同じ設計図でも仕上がりが微妙に変わるから
  解説: Chrome と Safari は別々の「大工」です。同じ設計図を渡しても、それぞれの得意分野や癖によって仕上がりに微妙な差が出ます。
  設計図（データ）自体は同じなので、「設計図が違う」わけではありません。
  Q5
  問題: この図のように、ブラウザはサーバーからデータを受け取っている。
  受け取ったデータに対して、ブラウザは次に何をする？
  正解: A. データを読み解いて、画面に組み立てる
  解説: ブラウザは受け取ったデータを読み解いて画面に組み立てています。
  サーバーはデータを届ける係、ブラウザはそれを形にする「大工」です。
  Q6（穴埋め）
  問題: Chrome、Safari、Edge など、サーバーからデータを受け取って画面に変換するアプリケーションを総称して何と呼ぶ？
  正解: ブラウザ
  解説: Chrome、Safari、Edgeなどの正式な総称はブラウザです。
  「browse」は英語で「見て回る」という意味。Webの世界を見て回るためのアプリケーションです。
  Q7
  問題: 大工（ブラウザ）はサーバーから届いた設計図（データ）を「読み解く」。
  具体的に何をしている？
  正解: A. 文字の大きさ・色・配置を計算して画面に並べること
  解説: ブラウザは設計図を読み解いて、文字の大きさ・色・配置を計算し、画面を組み立てています。
  この「設計図」の正体はHTML、CSS、JavaScriptです。
  Q8
  問題: Chrome、Safari、Edge…ブラウザがたくさんあるのは、なぜ？
  正解: A. より速く・使いやすくしようと各社が競争しているから
  解説: Google・Apple・Microsoftなど、各社が「もっと速く、もっと使いやすく」と競い合って独自のブラウザを開発しています。
  どのブラウザでも同じサイトを見られますが、大工が違うので仕上がりに個性が出ます。
  Q9
  問題: ブラウザは「設計図を読む大工」でした。この設計図の中身を知るために、まず理解したいのは？
  正解: A. 設計図（ソースコード）の中身と書き方
  解説: 大工（ブラウザ）の仕組みが分かったなら、次は設計図の中身を知る番です。
  HTML、CSS、JavaScript——これらが設計図の「書き方」であり、Webの世界をさらに深く理解する鍵になります。
  Q10（マッチング）
  問題: ブラウザの世界を「家づくり」に例えたとき、それぞれ何に当たる？
  正解:
  サーバーから届くデータ ↔ 設計図
  ブラウザ ↔ 大工
  表示された画面 ↔ 完成した家
  解説: サーバーから届くのは設計図、それを読んで組み立てるのが大工（ブラウザ）、出来上がった家が画面です。
  毎日ブラウザを使うたびに、この「家づくり」が一瞬で行われています。
  Lesson 4: URLの秘密
  Q1
  問題: ブラウザのアドレスバーに表示されている長い文字列。あの文字列にはどんな情報が入っている？
  正解: A. Webページの「住所」——場所を示す情報が全部詰まっている
  解説: アドレスバーの文字列はURLと呼ばれ、Webページの「住所」です。
  普段は気にせずコピペしているかもしれませんが、あの1文字1文字に「どこにあるページか」という情報が詰まっています。
  Q2（◯×）
  問題: 住所がなければ手紙が届かないのと同じように、URLがなければブラウザは裏側でWebページの場所を特定できない。
  正解: 正しい
  解説: 検索結果をクリックするときも、裏側ではブラウザがURLを頼りにサーバーを探してデータを受け取っています。
  住所なしに手紙を届けられないのと同じで、URLがなければ「どこにあるページか」が分かりません。IPアドレスで直接アクセスする場合もURL形式の一種です。
  Q3
  問題: URLは、Webページについてどんな情報を示している？
  正解: A. ページがある「場所」——どのサーバーのどのページか
  解説: URLが示すのはページの「場所」です。どのサーバーのどのページかという情報が、あの1行の文字列に詰まっています。
  住所が「建物の場所」を示すように、URLは「Webページの場所」を示します。
  Q4
  問題: URLの3つのパーツは、住所に例えるとそれぞれ何を示している？
  正解: A. 「届け方の指定」「建物名」「部屋番号」
  解説: URLの3パーツはそれぞれ住所の一部に対応しています。
  https://は届け方の指定（安全な届け方を使う宣言）、example.comは「どの建物か」、/aboutは「何階の何号室か」。現在のWebではhttps（安全な方式）がほぼ標準です。
  Q5
  問題: 「https://」と「http://」の違いは、あるアルファベット1文字。この「s」は何を意味している？
  正解: A. secure（安全）——暗号化して通信する
  解説: 「s」はsecure（安全）の頭文字です。httpsはデータを暗号化して安全にやり取りする方式です。
  このhttps://の部分は「プロトコル」と呼ばれ、通信のルールを指定しています。
  Q6（穴埋め）
  問題: https://の後ろに続く「example.com」のようなパーツは、住所でいう「建物名」に当たる。このパーツを何と呼ぶ？
  正解: ドメイン
  解説: ドメインはURLの中で「どの建物（サーバー）か」を示す部分です。
  google.com、youtube.com など、サイトごとに固有のドメインが割り当てられています。
  Q7
  問題: URLの「◯◯◯」の部分（/about）は、住所でいうと何に当たる？
  正解: A. 建物の中の「部屋番号」——どのページかを指定する
  解説: URLの末尾にあるスラッシュ以降の部分はパスと呼ばれます。
  建物（ドメイン）の中の「何階の何号室か」を指定する部分で、/about なら「このサイトについて」のページ、/contact なら「お問い合わせ」ページに案内されます。
  Q8
  問題: URLの正式名称「Uniform Resource Locator」の末尾にある「Locator」。この単語が意味することは？
  正解: A. 場所を特定するもの
  解説: Locatorは「場所を特定するもの」という意味です。
  URL全体では「統一された方法で、資源（ページ）の場所を特定するもの」。名前のとおり、URLの最大の役割は「場所の特定」です。
  Q9
  問題: 友人からURLが送られてきた。クリックする前に、URLのどこを見れば「どのサイトに飛ぶか」が分かる？
  正解: A. ドメイン（建物名）の部分
  解説: ドメインは「どの建物（サーバー）か」を示す部分でした。
  たとえば「amazon.com」と「amaz0n-login.com」ではまったく別の建物です。URLの構造を知っていれば、クリック前に行き先を確認できます。
  Q10（マッチング）
  問題: URLの各パーツと、住所での役割を正しく対応させよう。
  正解:
  https:// ↔ 届け方の指定
  example.com ↔ 建物名（サーバー）
  /about ↔ 部屋番号（ページ）
  解説: https://は届け方の指定、example.comは建物名（サーバー）、/aboutは部屋番号（ページ）です。
  この3パーツでWebページの「住所」が完成します。ブラウザはこの住所を使ってサーバーにデータを要求しています。
  Lesson 5: 注文と料理——リクエストとレスポンス
  Q1
  問題: ブラウザでWebページを開いたとき、そのページのデータはどこから来ている？
  正解: A. インターネットの向こう側にある別のコンピュータから届いた
  解説: Webページのデータは、インターネットの向こう側にある別のコンピュータ（サーバー）から届いています。
  ブラウザは表示するたびに、サーバーにデータを「もらいに行っている」のです。
  Q2（◯×）
  問題: URLを入力してEnterを押すと、ブラウザは裏で別のコンピュータとやり取りをしている。
  正解: 正しい
  解説: ブラウザはURLをもとにサーバーという別のコンピュータにデータを要求し、届いたデータを画面に表示しています。
  「読み込み中...」の間は、まさにこのやり取りが進行中です。
  Q3
  問題: ブラウザでWebページを開くとき、「ページをください」とお願いしているのは誰？
  正解: A. ブラウザ——サーバーに「このページをください」と要求する
  解説: 「ページをください」とお願いしているのはブラウザです。ブラウザがサーバーに要求を送り、サーバーがデータを返す——この「お願い」と「返答」の往復がWeb表示の基本です。
  この「お願い」のことを、次の問題でWeb用語として学びます。
  Q4
  問題: この図の2本の矢印は、それぞれ何を表している？
  正解: A. ブラウザからの「お願い」と、サーバーからの「返答」
  解説: 1本目はブラウザからサーバーへの「お願い」、2本目はサーバーからブラウザへの「返答」です。
  この往復が1回のWeb表示の正体。レストランの「注文→料理」と同じ流れです。
  Q5（穴埋め）
  問題: ブラウザからサーバーへの「お願い」を、Web用語では何と呼ぶ？
  正解: リクエスト
  解説: ブラウザからサーバーへのお願いはリクエスト（request）と呼ばれます。
  レストランの「注文」にあたるもので、「このページのデータをください」という要求のことです。
  Q6（穴埋め）
  問題: サーバーからブラウザへの「返答」を、Web用語では何と呼ぶ？
  正解: レスポンス
  解説: サーバーからブラウザへの返答はレスポンス（response）と呼ばれます。
  レストランの「料理」にあたるもので、リクエストに対してサーバーが返すデータのことです。
  Q7
  問題: ブラウザがリクエストを送るとき、「どのサーバーのどのページか」を伝えるために使うものは？
  正解: A. URL——サーバーとページの住所を指定する
  解説: リクエストを送るとき、ブラウザはURLを使って「どのサーバーのどのページか」を伝えます。
  手紙に住所がないと届かないのと同じで、URLがリクエストの「宛先」として機能しています。
  Q8（◯×）
  問題: サーバーから届いたレスポンスのデータは、そのまま画面に表示される。
  正解: 間違い
  解説: レスポンスで届いたデータは、ブラウザが解釈して画面に組み立てる工程を経て初めて表示されます。
  届いた料理をブラウザがお皿に盛り付けて完成させるイメージです。
  Q9
  問題: 検索、SNS、動画再生。これらの操作に共通する仕組みは何？
  正解: A. どれもリクエストとレスポンスのやり取りで動いている
  解説: 検索も、SNSも、動画再生も、すべてリクエスト→レスポンスの繰り返しです。
  操作の内容は違っても、裏側の仕組みは同じ。ブラウザがサーバーに「お願い」し、サーバーが「返答」する会話が何度も行われています。
  Q10（マッチング）
  問題: リクエストとレスポンスを、レストランの例えと正しく対応させよう。
  正解:
  リクエスト ↔ お客さんの注文
  レスポンス ↔ 厨房からの料理
  URL ↔ 届け先の住所
  解説: リクエストは注文、レスポンスは料理、URLは届け先の住所にあたります。
  この3つがセットになって、Web上の「やり取り」が成り立っています。仕組みが分かると「ページが表示されない」理由も想像できるようになります。
  Lesson 6: お客さんと厨房——クライアントとサーバー
  Q1
  問題: スマホでもパソコンでも、同じWebサイトが表示されます。
  このデータは、どこにあるのでしょう？
  正解: A. スマホやパソコンの中ではなく、別の場所にある
  解説: Webサイトのデータは、あなたの端末には入っていません。インターネットの先にある専用のコンピュータが保管していて、必要なときに届けてくれます。
  だから、どの端末からでも同じサイトが見られるのです。
  Q2（◯×）
  問題: 同じWebサイトでも、パソコンとスマホでは画面の見た目が違うことがある。これはデータの届き方に原因がある。
  正解: 間違い
  解説: サーバーから届くデータ自体は同じです。見た目が変わるのは、画面の大きさやブラウザの表示方法が違うからです。
  「データが違うから見た目が違う」のではなく、「同じデータを端末ごとに異なる方法で表示している」のが正体です。画面サイズに応じた表示切替（レスポンシブデザイン）もCSS側の仕組みであり、届くデータは同じです。
  Q3
  問題: レストランでは、お客さんが注文し、厨房が料理を準備します。
  Webの世界でも同じ「役割分担」があります。お客さんにあたるのは？
  正解: A. ブラウザ
  解説: レストランのお客さんが注文するように、ブラウザがサーバーに「このページをください」とお願いします。
  ブラウザは「注文する側」、サーバーは「料理を準備する側」です。
  Q4
  問題: この図のやり取りで、最初に行動を起こすのはどちらですか？
  正解: A. お客さん（左側）が先に注文する
  解説: 最初に動くのはお客さん（クライアント）です。お客さんが注文（リクエスト）しなければ、厨房（サーバー）は何も届けません。
  Webでも同じで、ブラウザが先にお願いしないとサーバーは動きません。
  Q5（穴埋め）
  問題: Webの世界では、注文する側（ブラウザ）を専門用語で何と呼ぶでしょうか？
  正解: クライアント
  解説: 注文する側をクライアントと呼びます。英語の client から来た言葉で、もともと「依頼人」という意味です。
  ブラウザやスマホアプリは、サーバーに「お願い」をする側なので、クライアントにあたります。
  Q6（穴埋め）
  問題: レストランの厨房にあたる、Webの裏方を何と呼ぶでしょうか？
  正解: サーバー
  解説: 厨房にあたるのはサーバーです。英語の serve（提供する）から来た言葉で、「データを提供する側」という意味です。
  リクエストを受け取り、必要なデータを準備して返すコンピュータがサーバーです。
  Q7
  問題: あなたがスマホで検索ボタンを押しました。
  このとき最初に起きることは？
  正解: A. ブラウザがサーバーにリクエストを送る
  解説: 検索ボタンを押すと、まずブラウザがサーバーにリクエスト（お願い）を送ります。サーバーはそれを受け取って、検索結果をレスポンス（返答）として返します。
  検索もSNSも動画も、すべてこの「お願い→返答」の繰り返しです。
  Q8
  問題: 「サーバー」とは、どのようなものですか？
  正解: A. リクエストを受けてデータを返す役割を持つコンピュータ
  解説: サーバーとは特別な機械のことではなく、リクエストを受けてデータを返す「役割」のことです。
  実際には24時間動き続ける専用のコンピュータが使われますが、仕組み自体は普通のパソコンでも同じです。
  Q9
  問題: Webの世界では「画面に表示される内容」はどこから届きますか？
  正解: A. サーバーからクライアントに届く
  解説: 画面に表示される内容の元データは、サーバーから届きます。ブラウザはその届いたデータを使って画面を組み立てますが、データの出どころはサーバーです。
  クライアント（ブラウザ）がリクエストを送り、サーバーがレスポンスとしてデータを返す——この役割分担がWebの基本です。
  Q10（マッチング）
  問題: レストランの例えと、Webの用語を正しく対応させてください。
  正解:
  お客さん ↔ クライアント
  厨房 ↔ サーバー
  注文 ↔ リクエスト
  料理 ↔ レスポンス
  解説: お客さん＝クライアント、厨房＝サーバー、注文＝リクエスト、料理＝レスポンス。
  この4つの対応を覚えておけば、Webの通信の仕組みを人に説明できるようになります。
  Lesson 7: ブラウザの先にあるWeb
  Q1
  問題: 天気アプリを開くと、最新の天気予報が表示されます。
  この天気データは、どこから届いていますか？
  正解: A. インターネットの向こうにあるサーバーから届いている
  解説: 天気アプリは裏側でサーバーにリクエストを送って最新データを受け取っています。
  アプリを開くたびに、インターネットを通じてやり取りが起きているのです。
  Q2（◯×）
  問題: SNSアプリの新着通知が届くのは、投稿データがスマホの中に保存されているからである。
  正解: 間違い
  解説: 新着情報はサーバーからスマホに届けられたものです。
  スマホの中にデータがあるのではなく、サーバーと通信して初めて新しい投稿を知ることができます。
  Q3
  問題: サーバーを厨房に例えると、ブラウザもスマホアプリも「お客さん」です。
  この「お客さん」の役割を、何と呼びますか？
  正解: A. クライアント
  解説: ブラウザもスマホアプリも、サーバーにリクエストを送る側という点で同じ「お客さん」です。
  この役割をクライアントと呼びます。
  Q4
  問題: ブラウザとスマホアプリ、裏側の仕組みで共通しているのは？
  正解: A. サーバーにリクエストを送り、レスポンスを受け取る仕組み
  解説: 見た目や使い方は違っても、裏側では同じ「リクエスト→レスポンス」の仕組みが動いています。
  これがWeb技術の基本パターンです。
  Q5
  問題: ブラウザはURLを入力してWebサイトを開きます。
  では天気アプリは、どうやってサーバーと通信していますか？
  正解: A. アプリが裏側で自動的にリクエストを送っている
  解説: ブラウザはURLを入力し、アプリはアイコンをタップ。操作方法は全く違います。
  しかし裏側では、どちらもアプリが自動的にサーバーへリクエストを送る同じ仕組みが動いています。
  Q6
  問題: 天気アプリ、地図アプリ、SNSアプリ。これら3つに共通することは？
  正解: A. すべて裏側でサーバーにリクエストを送っている
  解説: 見た目も使い方もバラバラですが、裏ではどれもサーバーとやり取りしています。
  この共通パターンが「Web技術」の正体です。
  Q7（◯×）
  問題: 電波が届かない場所では、スマホアプリでも新しい天気情報を取得できない。
  正解: 正しい
  解説: 天気アプリはサーバーにリクエストを送ってデータを受け取る仕組みです。
  電波がなければサーバーと通信できないので、新しい情報は届きません。
  Q8（マッチング）
  問題: アプリが裏側でサーバーから受け取っているデータを対応させてください。
  正解:
  天気アプリ ↔ 最新の気温と降水確率
  地図アプリ ↔ 現在地の周辺情報
  SNSアプリ ↔ 友達の新着投稿
  解説: 受け取るデータの種類はアプリごとに違いますが、どれも「サーバーにお願い→データをもらう」という流れは同じです。
  Q9
  問題: リクエスト→レスポンスの仕組みは、どの範囲で使われている技術ですか？
  正解: A. ブラウザやアプリなど多くのサービスの共通基盤
  解説: Web技術（リクエスト→レスポンス、クライアント→サーバー）は、多くのデジタルサービスの共通基盤です。
  ブラウザだけでなく、スマホアプリやさまざまなサービスの裏側で使われています。
  Q10（穴埋め）
  問題: 友達に「天気アプリってどうやって最新情報を知るの？」と聞かれました。
  正解: サーバー
  解説: 天気アプリもブラウザも、裏側ではサーバーにリクエストを送ってデータを受け取る仕組みです。
  新しい情報が欲しいときは、必ずサーバーとやり取りする必要があります。
  Lesson 8: Webの地図を完成させよう（The Finale）
  Q1
  問題: URLを入力してEnterを押すと、ページが表示される。
  この一瞬で何が起きていますか？
  正解: A. サーバーにリクエストを送り、データで画面を組み立てている
  解説: ブラウザがURLをもとにサーバーへリクエストを送り、届いたデータを画面に組み立てて表示しています。
  この「注文→受け取り→組み立て」が毎回起きています。
  Q2（◯×）
  問題: インターネットとWebは同じものである。
  正解: 間違い
  解説: インターネットは世界中のコンピュータを繋ぐ道路網です。Webはその道路の上を走るサービスの1つ。
  メールやビデオ通話もインターネットを使いますが、Webではありません。
  Q3
  問題: レストランの例えで「厨房」にあたるWeb用語はどれですか？
  正解: A. サーバー
  解説: 料理を作る「厨房」にあたるのがサーバーです。お客さんの注文（リクエスト）に応じて、データ（料理）を準備して返します。
  お客さん＝クライアント、注文＝リクエスト、料理＝レスポンスも合わせて覚えましょう。
  Q4
  問題: サーバーからブラウザに届いた「料理」。
  この料理の正体は何でしょう？
  正解: A. Webページの材料データ
  解説: 届く「料理」の正体はWebページの材料データです。ブラウザがこの材料を受け取り、画面に仕上げています。
  この材料の中身がHTML・CSS・JavaScriptという3種類の設計図です。
  Q5
  問題: Webの全体像を下から見ていくと、一番の土台にあるのは何ですか？
  正解: A. インターネット（道路網）
  解説: 一番の土台はインターネット（道路網）です。この道路の上にWebがあり、ブラウザがURLを使ってリクエスト/レスポンスをやり取りしています。
  インターネットがなければ、Webは存在できません。
  Q6
  問題: ブラウザの役割として最も正確なのはどれですか？
  正解: A. サーバーからデータを受け取り、画面に組み立てるアプリ
  解説: ブラウザは「設計図を読む大工」です。サーバーから届いたデータ（設計図）を読んで、画面（家）に組み立てます。
  ChromeやSafariなどのアプリがこの仕事をしています。
  Q7（マッチング）

  解説: 住所＝URL、お客さん＝クライアント、注文＝リクエスト、厨房＝サーバー。この4つの対応関係がWebの基本です。

  Q8

  Q9（穴埋め）

  正解: ブラウザ／サーバー
  解説: ブラウザがURL（住所）をもとにサーバーにリクエストを送り、届いたデータを画面に組み立てる——これがWebの基本の仕組みです。
  この一文で説明できるようになれば、Webの地図は完成です。

  Lesson 1: Webページの裏側を覗いてみよう
  Q1
  問題: 普段見ているWebページのきれいな画面。

  解説: Webページの裏側には、文字で書かれた指示書が隠れています。「ここに見出しを置いて、この色で背景を塗って」といった指示です。

  Q7

  Q8（マッチング）

  Lesson 2: 設計図の正体——ソースコードとは
  Q1
  問題: Webページの裏側にある「文字の並び」。実は、メモ帳で開くことができます。
  これはどういう意味でしょう？
  正解: A. ソースコードはただのテキストファイルである
  解説: ソースコードはただのテキストファイルです。特別なソフトがなくても、メモ帳で読み書きできます。
  難しそうに見えても、中身は文字の並び。「自分にも開ける」と知るだけで、ハードルがぐっと下がりますね。
  Q2
  問題: Webサイトを作る時、ファイルにはいくつかの種類があります。
  .html、.css、.js に共通する特徴は何でしょう？
  正解: A. どれもメモ帳で開ける普通のテキストファイル
  解説: .html、.css、.js はどれもテキストファイルです。末尾の名前（拡張子）は違いますが、中身はすべて文字で書かれています。
  拡張子が違うだけで、中身はどれも「文字の並び」。これがソースコードの正体です。
  Q3
  問題: ソースコードを「家の設計図」に例えると、設計図が書かれている「紙」にあたるのは何でしょう？

  解説: 設計図（ソースコード）が書かれているのはただのテキストファイル——特別な素材ではありません。
  紙に書かれた設計図を大工が読むように、テキストファイルに書かれたコードをブラウザ（大工）が読み取って画面を組み立てています。
  Q4
  問題: テキストファイル→ブラウザ→Webページ。この流れでブラウザが果たしている役割はどれでしょう？
  正解: A. テキストを読み取って画面に変換する「大工」
  解説: ブラウザはテキストを画面に変換する「大工」です。サーバーから届くのは完成した画面ではなく、ただのテキスト。
  ブラウザがそのテキストを読み取り、色やレイアウトを付けて、あなたが見る画面に組み立てています。
  Q5（穴埋め）
  問題: Webページのソースコードの正体は、特殊な形式のファイルではありません。
  正解: テキストファイル
  解説: ソースコードはテキストファイル——文字だけで構成されたファイル形式です。
  .htmlや.cssのように拡張子は違っても、中身はどれも文字の並び。だからメモ帳でも開けるのです。
  Q6（◯×）
  問題: ブラウザは受け取ったソースコードを、一度すべて読み終えてから画面を表示し始める。
  正解: 間違い

  Q7（マッチング）

  .html ↔ ページの構造を定義
  .css ↔ ページの見た目を定義
  .js ↔ ページの動きを定義
  解説: .htmlは構造、.cssは見た目、.jsは動き。3種類の設計図が、それぞれの専門分野を担当しています。

  Q8
  問題: 物理の家は設計を変えると建て直しに何ヶ月もかかります。ソースコードを書き直した場合、ブラウザはどうしますか？

  Q9
  問題: ソースコードがテキストファイルだと分かりました。

  正解: A. 実際にソースコードを見てみる
  解説: ただのテキストなら、まず見てみるのが一番です。特別な準備は要りません。
  ブラウザには「開発者ツール」という裏側を覗く機能が最初から入っています。この先で体験しましょう。
  Q10
  問題: 設計図（ソースコード）には3種類のファイルがあると分かりました。
  家に例えると、この3つはそれぞれ何にあたるでしょう？
  正解: A. 骨組み・内装・家電
  解説: 骨組み（構造）・内装（見た目）・家電（動き）——3種類の設計図が、それぞれ専門の仕事を担当しています。
  骨組みがなければ壁も家具も置けないように、構造を定義するファイルが土台です。これから1つずつ解剖していきます。
  Lesson 3: 骨組みの担当——HTMLという技術
  Q1
  問題: もし家に骨組み（柱や壁）がなかったら、どうなるでしょう？
  正解: A. 家具もドアも設置する場所がなく、何も始まらない
  解説: 柱や壁がなければ、家具を置く場所もドアをつける場所もありません。骨組みがすべての土台です。
  Webページにも同じことが言えます。「見出し」「段落」「リスト」という骨組みがなければ、色をつけたり動きをつけたりすることもできません。
  Q2
  問題: CSS（見た目を整える技術）もJavaScript（動きをつける技術）も使わず、HTMLだけで作ったページはどんな見た目になるでしょう？
  正解: A. 白い背景に黒い文字が並ぶだけの素朴なページ
  解説: HTMLだけの世界は、白い背景に黒い文字が並ぶだけの素朴な姿です。華やかな色やレイアウトの指定はありません。
  ここにCSS（内装）を加えると色やレイアウトが整い、JavaScript（家電）を加えるとボタンやメニューが動き出します。骨組みだけの家があるからこそ、後から飾れるのです。
  Q3
  問題: 家の骨組みが「柱と壁で構造を決める」ように、Webページにとっての「骨組み」にあたる技術はどれですか？
  正解: A. ページの構造（見出し・段落・リスト）を決める技術
  解説: 「骨組み」にあたるのは、ページの構造を決める技術です。どこが見出しで、どこが段落で、どこがリストかを定義します。
  色やレイアウトは「内装」、動きは「家電」の役割。サーバー送信は構造とは別の仕事です。
  Q4
  問題: 家の設計図に「ここはリビング」「ここは寝室」と部屋の名前を書くように、HTMLはテキストに何をしますか？
  正解: A. 「ここは見出し」「ここは段落」と意味の印をつける
  解説: HTMLは、テキストに「ここは見出し」「ここは段落」と意味の印（マーク）をつける技術です。この「印をつける」ことを英語で「Markup」と呼びます。
  色やサイズはCSS、動きはJavaScriptの仕事。HTMLは「構造の設計図」に徹します。
  Q5
  問題: Webページを構成する3つの技術を、家に例えた図です。なぜHTMLが一番下（土台）に置かれているのでしょう？
  正解: A. 骨組みがないと内装も家電も設置できないから
  解説: HTMLが一番下にあるのは、骨組みがすべての土台だから。構造がなければ、色を塗る対象も動きをつける対象もありません。
  3つの技術が積み重なって、はじめて完成したWebページになります。
  Q6（穴埋め）
  問題: HTMLの正式名称を完成させましょう。「Markup」は「印をつける」という意味です。
  正解: HyperText
  解説: HTMLは HyperText Markup Language の略です。「HyperText」は「文書同士がつながる（リンクする）テキスト」という意味。
  「Markup」は「印をつける」。つまりHTMLは「つながるテキストに、構造の印をつける言語」です。
  Q7
  問題: HTMLの設計図には head（設定の区画）と body（本文の区画）があります。
  画面に表示される内容（見出し・段落・画像など）を置く場所はどちらですか？
  正解: A. body（本文の区画）
  解説: 画面に表示される内容（見出し、段落、画像など）はbody（本文の区画）に置きます。
  もう一方のhead（設定の区画）にはページのタイトルや裏方の情報が入ります。家で言えば「住む空間」と「表札や契約書」の違いです。
  Q8（◯×）
  問題: HTMLの骨組みが正確でなくても、CSSで見た目を整えれば問題なく使えるページになる。
  正解: 間違い
  解説: これは間違いです。CSSは骨組みに装飾を施す技術なので、HTMLの構造が正確でなければ意図通りに装飾できません。
  柱が傾いた家に壁紙を貼ってもきれいに仕上がらないのと同じです。
  Q9
  問題: HTMLで骨組みをしっかり作っておくと、どんなメリットがありますか？
  正解: A. 同じ骨組みにCSSで異なる見た目を着せ替えできる
  解説: 骨組み（HTML）がしっかりしていれば、CSSを変えるだけで見た目をガラリと変えられます。
  スマホの画面を暗くする「ダークモード」も着せ替えの一種。構造と見た目が分かれているからこそ、自由自在に装飾を変えられるのです。
  Q10
  問題: CSSやJavaScriptが正しく機能するために、必ず先に必要な土台はどれですか？
  正解: A. HTML（構造）
  解説: HTML（骨組み）がすべての土台です。柱を立てなければ壁紙は貼れませんし、家電も置けません。
  骨組みの構造を知ることが、Webページを「読み解ける人」への第一歩です。
  Lesson 4: 内装の担当——CSSという技術
  Q1
  問題: HTMLだけで作ったWebページを見ると、普段のサイトと何かが違います。
  普段のサイトと比べて、決定的に足りないものは何でしょう？
  正解: A. 見た目を整える仕組み
  解説: HTMLは「構造」だけを定義する言語なので、色やレイアウトの機能を持っていません。普段のサイトが華やかなのは、見た目を整える別の技術が加わっているからです。
  Q2（◯×）
  問題: HTMLだけのWebページでも、文字情報を読み取るには十分である。
  正解: 正しい
  解説: 骨組みだけの家でも部屋の区切りはわかるように、HTMLだけのページでも文字情報は読み取れます。
  ただし見た目を整えるには「内装」にあたる技術が必要です。
  Q3
  問題: 骨組み（HTML）が決まった家に、壁紙を貼り、家具を置き、照明を整える。この「内装」にあたるWebの技術はどれ？
  正解: A. CSS
  解説: CSS（Cascading Style Sheets）が「内装・インテリア」の役割です。
  骨組み（HTML）はそのままに、壁紙（色）を変えたり家具（レイアウト）を配置したりして、見た目を整えます。
  Q4
  問題: 同じ間取り（HTML）の部屋でも、内装を変えるだけで和風にも北欧風にもなります。Webページでこの「着せ替え」を実現しているのは？
  正解: A. CSS
  解説: CSSが「着せ替え」の正体です。骨組み（HTML）はそのままで、CSSファイルを差し替えるだけで見た目がガラッと変わります。
  和風・北欧風・モダンと、内装（CSS）次第で雰囲気が一変するのです。
  Q5（穴埋め）
  問題: Webページの見た目を定義する言語の名前は何でしょう？
  正解: CSS
  解説: CSS（Cascading Style Sheets）のCascadingは「上から順にスタイルが伝わる」という意味です。色・サイズ・配置・フォントなど、見た目に関わるすべてを担当します。
  Q6
  問題: HTMLが「何があるか（構造）」を定義するなら、CSSは何を定義する技術でしょう？
  正解: A. どう見えるか（見た目）
  解説: CSSは「どう見えるか」（見た目）を定義します。色・サイズ・配置・フォントなど、見た目に関わる指定はすべてCSSの仕事です。
  構造の定義はHTML、動作の定義はまた別の技術が担当します。
  Q7（マッチング）
  問題: HTMLとCSSの担当を正しく結びつけてください。
  正解:
  見出しや段落を定義する ↔ HTML
  文字の色を変える ↔ CSS
  箇条書きの構造を作る ↔ HTML
  要素の配置を決める ↔ CSS
  解説: HTMLは「何があるか」（構造）を定義し、CSSは「どう見えるか」（見た目）を定義します。見出しや箇条書きはHTML、色や配置はCSSの仕事です。
  Q8
  問題: CSSだけを用意しても、HTMLがなければWebページの見た目は作れません。なぜでしょう？
  正解: A. 装飾する対象（骨組み）がないから
  解説: 内装（CSS）を施すには、まず骨組み（HTML）が必要です。壁紙を貼るにも壁がなければ貼れません。
  HTMLが土台、CSSはその上に乗る装飾という依存関係を覚えておきましょう。
  Q9
  問題: スマホの「ダークモード」では背景が暗くなり文字が白くなります。この色の変更は、主にどの技術の担当でしょう？
  正解: A. 主にCSSの担当
  解説: ダークモードで変わるのは色や背景の指定で、これはCSSの担当です。
  切り替えボタンの仕組みは別の技術が支えますが、見た目の変更そのものはCSSの仕事です。
  Q10
  問題: CSSは「家を建てる」たとえで言うと、どの役割でしょうか？
  正解: A. 内装・インテリア（壁紙・家具・照明）
  解説: CSSは内装・インテリアです。骨組み（HTML）が「何があるか」を決め、内装（CSS）が「どう見えるか」を整えます。
  3つの技術のうち2つ（骨組みと内装）の役割が見えてきました。
  Lesson 5: 家電の担当——JavaScriptという技術
  Q1
  問題: 骨組み（HTML）と内装（CSS）が揃った家。見た目はきれいだけど、ボタンを押しても何も起きない。
  この家に足りないのは？
  正解: A. 家電（動く仕組み）
  解説: 足りないのは家電（動く仕組み）です。骨組みと内装だけでは、家に住めても照明やエアコンは動きません。
  Webページも同じ。HTMLとCSSだけでは情報は読めますが、ボタンを押して何かが起きる——そんな「動き」は出せないのです。
  Q2（◯×）
  問題: HTMLとCSSだけでは、ボタンを押してサーバーからデータを取得したり、表示する文字や数値をプログラムで書き換えたりする処理はできない。
  正解: 正しい
  解説: CSSの:hoverや:activeでボタンの見た目を変えることはできますが、データを取得したり、計算結果を表示したりする「プログラム的な処理」には別の技術が必要です。
  その技術こそ、これから学ぶ「家電」担当です。
  Q3
  問題: Webページに「動く仕組み」を追加する技術の名前はどれですか？
  正解: A. JavaScript
  解説: JavaScriptがWebページの「家電」担当です。ユーザーがボタンを押す→メニューが開く。文字を入力する→候補が表示される。
  こうした「操作に反応して画面が変わる」仕組みは、すべてJavaScriptの仕事です。
  Q4
  問題: 検索サイトで文字を入力すると、候補がリアルタイムで表示されます。
  この機能を実現しているのはどの技術？
  正解: A. JavaScript
  解説: 入力に応じて候補をリアルタイム表示する——こうしたユーザーの操作に即座に反応する機能はJavaScriptの仕事です。
  HTMLは「検索窓がある」という構造、CSSは「検索窓の見た目」を担当しています。
  Q5
  問題: Webページは3つの技術が積み重なってできています。それぞれの担当として正しいのはどれ？
  正解: A. HTML＝構造、CSS＝見た目、JavaScript＝動き
  解説: HTML＝構造、CSS＝見た目、JavaScript＝動き。3つの技術がそれぞれ別の担当を持ち、土台から順に積み重なっています。
  骨組みがなければ内装は貼れず、内装がなくても家電は動きますが、骨組みは必須です。
  Q6（穴埋め）
  問題: 構造・見た目・動きのうち、JavaScriptが担うのはどれ？
  正解: 動き
  解説: JavaScriptの担当は「動き」です。ボタンを押したらメニューが開く、スクロールしたら画像が現れる——画面上で「何かが変化する」仕組みを作ります。
  構造はHTML、見た目はCSSの担当です。
  Q7
  問題: HTML・CSS・JavaScriptの3つの技術は、それぞれ別のファイルに書かれています。
  ボタンを押したときの動作を担当しているのはどのファイル？
  正解: A. script.js（JavaScriptファイル）
  解説: ボタンの動作を担当するのはscript.js（JavaScriptファイル）です。
  index.htmlが「ボタンがここにある」を定義し、style.cssが「ボタンの見た目」を整え、script.jsが「押されたら何をするか」を決めています。
  Q8（マッチング）
  問題: 3つの技術と「家を建てる」たとえを正しく結びつけてください。
  正解:
  HTML ↔ 骨組み
  CSS ↔ 内装
  JavaScript ↔ 家電
  解説: HTML＝骨組み（ページの構造）、CSS＝内装（見た目の装飾）、JavaScript＝家電（操作に反応する動き）。
  ボタンを押したらメニューが開く、入力したら候補が出る——こうした動的な反応がJavaScriptの実際の仕事です。
  Q9（並び替え）
  問題: ブラウザがWebページを組み立てる順番を正しく並べてください。
  正解: HTML → CSS → JavaScript
  解説: ブラウザはまずHTML→CSS→JavaScriptの順で画面を組み立てます。骨組みを作り、見た目を整え、最後に動きを加える。
  骨組みがなければ内装も家電も設置できない。だからHTMLが最初なのです。
  Q10
  問題: HTML＋CSS＋JavaScriptの3つが揃った状態を、「家を建てる」メタファーで表すとどうなりますか？
  正解: A. 骨組み・内装・家電が揃って「住める家」が完成する
  解説: 骨組み・内装・家電がそれぞれ独立した役割を持ちながら、協力して「住める家」を完成させます。
  どれか1つが欠けても「住める家」にはならない——3つの技術が揃って初めてWebページは完成するのです。
  Lesson 6: なぜ3つに分かれているのか
  Q1
  問題: HTML・CSS・JavaScriptの3つの技術。全部1つのファイルにまとめた方がシンプルに思えます。
  実際にはどうでしょう？
  正解: A. 小さなページなら1つでも動くが、大きくなると管理が難しくなる
  解説: 実は1つのファイルにまとめても動きます。でも規模が大きくなると、どこに何が書いてあるか分からなくなるのが問題です。
  小さな家なら1人で建てられますが、ビルは専門家の分業が必要ですよね。Webも同じです。
  Q2
  問題: 家の設計図が1冊にまとまっています。骨組み・壁紙・配線の指示がすべて混ざって書いてあります。
  壁紙だけ変えたいとき、何が困りますか？
  正解: A. 壁紙を探すのに手間取り、他を壊すリスクがある
  解説: 全部が1冊にまとまっていると、壁紙の記述を探すだけで一苦労です。うっかり骨組みの指示を変えてしまうリスクもあります。
  Webも同じ。見た目の変更だけなのに、構造や動きのコードが混在していると大変です。
  Q3（マッチング）
  問題: 家づくりの専門家と、Web技術を正しく対応させてください。
  正解:
  骨組み職人 ↔ HTML
  内装屋（壁紙・家具） ↔ CSS
  電気屋（配線・家電） ↔ JavaScript
  解説: 骨組み職人＝HTML（構造）、内装屋＝CSS（見た目）、電気屋＝JavaScript（動き）。専門家が分業するから、変更したい部分だけ依頼できます。
  Q4（◯×）
  問題: HTML・CSS・JavaScriptがファイルごとに分かれているサイトがある。
  文字の色を赤から青に変えたいとき、HTMLファイルも書き直す必要がある。
  正解: 間違い
  解説: きちんと分離されていれば、文字の色は「見た目」担当のCSSだけで変更できます。HTMLは構造の担当なので触る必要がありません。
  もし1つのファイルに全部書いてあったら、こうはいきませんよね。分離のメリットです。
  Q5
  問題: 3つの技術のうち、「動きの定義」を担当するのはどれですか？
  正解: A. JavaScript
  解説: JavaScriptが「動き」の担当です。ボタンを押したら反応する、メニューが開くなど、ページに動きを加えるのが仕事です。
  「構造」はHTML、「見た目」はCSSがそれぞれ担当しています。
  Q6
  問題: Webサイトの背景や文字の色を暗い配色に変えたいとき、主にどの技術のファイルを変更すれば済みますか？
  正解: A. CSSだけ変更すれば済む
  解説: 色や背景の変更は「見た目」の領域なので、主にCSSだけの修正で実現できます。構造（HTML）や動き（JavaScript）を大きく変える必要はありません。
  これが「分けて管理する」ことの大きなメリットです。
  Q7（穴埋め）
  問題: 構造・見た目・動きをそれぞれ独立に管理する設計思想には、正式な名前があります。
  正解: 関心の分離
  解説: 関心の分離（Separation of Concerns）は、「構造」「見た目」「動き」をそれぞれ独立に管理する考え方です。
  変更の影響範囲を最小限に抑え、チームでの分業も可能にする設計思想です。
  Q8
  問題: 家を建てるとき、最初に作らなければならないのは？
  Webの3層構造にも同じ順番があります。
  正解: A. 骨組み（HTML）—— これがないと壁紙も家電も置けない
  解説: 家と同じで、まず骨組み（HTML）を立ててから、壁紙を貼り（CSS）、家電を設置します（JavaScript）。
  HTMLが「一番偉い」のではなく、組み立ての順番として最初に必要ということです。
  Q9
  問題: HTML・CSS・JavaScriptが分かれていると、作業するときにどんなメリットがありますか？
  正解: A. 変更したい部分だけに集中でき、他を壊す心配が少ない
  解説: 分かれているから、変更したい部分だけに集中できます。見た目を直したいならCSSだけ、動きを変えたいならJavaScriptだけ。
  1人でも複数人でも、「今日はここだけ」と作業範囲を絞れるのが大きなメリットです。
  Q10
  問題: 「関心の分離」という考え方。これはどの範囲で使われている知恵でしょう？
  正解: A. あらゆるものづくりに通じる普遍的な設計思想
  解説: 「関心の分離」はあらゆるものづくりに通じる普遍的な知恵です。
  映画でも「脚本」「撮影」「音楽」は専門家が分業します。それぞれの専門に集中できるから品質が上がる——Webも同じです。
  Lesson 7: 裏側を覗く道具——開発者ツール
  Q1
  問題: HTML・CSS・JSの概念は分かった。でも「本当にそうなの？」と自分の目で確かめたくなります。
  どうすれば確かめられるでしょう？
  正解: A. ブラウザに最初から入っている「裏側を覗く道具」を使う
  解説: 実はブラウザには開発者ツールという裏側を覗く道具が最初から内蔵されています。追加のソフトも、サーバーへの問い合わせも不要です。
  どの主要ブラウザにも入っている標準機能です。
  Q2（◯×）
  問題: 開発者ツールはGoogle Chromeだけの特別機能であり、SafariやEdgeには搭載されていない。
  正解: 間違い
  解説: いいえ。開発者ツールはすべての主要ブラウザに標準搭載されています。Chrome、Safari、Edge——どれを使っていても同じように裏側を覗けます。
  特定のブラウザ専用ではなく、どのブラウザでも使える共通の道具です。
  Q3
  問題: 開発者ツールは「家の壁を透明にする魔法のメガネ」に例えられます。このメガネで見えるものはどれですか？
  正解: A. 骨組み（HTML）や内装の仕様（CSS）など裏側の構造や設定
  解説: 開発者ツールを使うと、骨組み（HTML）や内装の仕様書（CSS）——つまり、画面の裏側に隠れている設計図を見通せます。
  サーバーのプログラムは手元には届かないため、このメガネでも見ることはできません。見えるのはブラウザに届いた「完成品の中身」だけです。
  Q4
  問題: 開発者ツールの「Elements」（要素）パネルでは、あるものとWebページの画面を対応づけて確認できます。
  何と対応づけられますか？
  正解: A. HTMLの構造（ソースコード）と画面の表示
  解説: ElementsパネルではHTMLの構造と画面の表示を対応づけて確認できます。
  「この見出しは、設計図のどの部分で定義されているのか」が一目で分かる仕組みです。
  Q5
  問題: ブラウザで開発者ツールを開く方法として正しいのはどれですか？
  正解: A. F12キー（または右クリック→「検証」）を選ぶ
  解説: F12キーを押すか、ページ上で右クリック→「検証」を選ぶのが一般的な開き方です。Macでは Cmd+Option+I でも開けます。
  インストール不要で、どのWebページを開いていても、その場ですぐ使えます。
  Q6
  問題: ソースコード（設計図）→ブラウザ（大工）→画面（家）という変換の流れがあります。開発者ツールはこの流れのどこを覗く道具ですか？
  正解: A. 設計図（ソースコード）の部分を覗く
  解説: 開発者ツールは設計図（ソースコード）を覗く窓です。完成した家（画面）の裏に隠れている、HTMLやCSSの設計図を直接確認できます。
  Networkなどのパネルではブラウザの通信も覗けますが、まずはElementsパネルから始めましょう。
  Q7
  問題: 開発者ツールのElementsパネルで、HTMLの構造だけでなく、もう1つ重要なものが確認できます。それは何ですか？
  正解: A. 各要素に適用されているCSS（見た目の設定）
  解説: Elementsパネルでは、HTML構造に加えて各要素に適用されているCSSも確認できます。「この文字の色はどこで指定されているのか」が一目で分かります。
  JavaScriptの全文は別の「Sources」パネルで確認します。Elementsは構造と見た目に特化した窓です。
  Q8（穴埋め）
  問題: ブラウザに内蔵された、Webページの裏側を確認するための道具。この道具の正式な日本語名は何でしょう？
  正解: 開発者ツール
  解説: 正式には開発者ツール（DevTools）と呼びます。「開発者」と名前がついていますが、Webに興味がある人なら誰でも使える道具です。
  HTMLの構造やCSSの設定を、自分の目で確認できます。
  Q9
  問題: 気になるWebサイトのデザインを見て「この色はどうやって指定しているんだろう？」と思いました。
  自分で調べるにはどうすればよいですか？
  正解: A. 開発者ツールでその要素のCSSを確認する
  解説: 開発者ツールを使えば、どんなWebサイトでもCSSの設定を自分で調べられます。
  文字の色、サイズ、余白——気になった設定をその場で確認できるのが開発者ツールの強みです。
  Q10（◯×）
  問題: 開発者ツールはプロの開発者だけが使う専門的なツールであり、学び始めの人には難しすぎる。
  正解: 間違い
  解説: いいえ。開発者ツールは学び始めの人にこそ役立つ道具です。「この画面の裏側はどうなっているのか？」を自分の目で確かめられます。
  まずはF12を押してElementsパネルを開いてみましょう。最初は全部分からなくても大丈夫です。
  Lesson 8: Webページの設計図を読み解こう
  Q1
  問題: 右クリックで「ページのソースを表示」を押すと、謎の文字列が現れました。
  あの「謎の文字列」の正体は何でしたか？
  正解: A. Webページの設計図（ソースコード）
  解説: あの文字列はWebページの設計図（ソースコード）です。ブラウザという「大工」がこの設計図を読んで、画面という「家」を組み立てています。
  最初は謎に見えたものの正体が分かった——この発見が、Webページの仕組みを理解する出発点でした。
  Q2
  問題: 設計図（ソースコード）が画面になるまでの流れで、ブラウザが担う役割はどれですか？
  正解: A. 設計図を読んで画面に組み立てる「大工」
  解説: ブラウザは設計図を読んで画面に組み立てる「大工」です。開発者が設計図を書き、サーバーが保管し、ブラウザが組み立てる——三者の分業で成り立っています。
  しかもこの大工は、開発者が設計図を書き直すと瞬時に建て直してくれます。
  Q3
  問題: 家を建てるには、まず何がないと始まりませんか？
  正解: A. 骨組み（HTML）
  解説: まず骨組み（HTML）がないと、内装も家電も設置できません。最終的には3つ揃いますが、組み立ての順序として骨組みが最初に必要です。
  HTMLが土台だからこそ、最初に深く学ぶ価値があります。
  Q4（◯×）
  問題: CSSを変更しただけで、HTMLの記述内容が書き換わる。
  正解: 間違い
  解説: CSSは見た目だけを担当しています。CSSを変えても、骨組み（HTML）には影響しません。
  これが「関心の分離」のメリットです。壁紙を張り替えても、柱は動かない。
  Q5（マッチング）
  問題: 3つの技術と「家を建てる」メタファーを正しく対応させてください。
  正解:
  HTML ↔ 骨組み
  CSS ↔ 内装
  JavaScript ↔ 家電
  解説: HTML＝骨組み、CSS＝内装、JavaScript＝家電。この3つの分業がWebページの基本です。
  この対応が頭に入っていれば、Webページの設計図を「読める」ようになります。
  Q6
  問題: HTML・CSS・JavaScriptをわざわざ別のファイルに分けている理由として、最も的確なのはどれですか？
  正解: A. 変更の影響を最小限に抑えるため
  解説: 3つに分けるのは変更の影響を最小限に抑えるためです。色を変えたいときはCSSだけ、動きを変えたいときはJavaScriptだけ修正すれば済みます。
  この考え方を「関心の分離」と呼びます。
  Q7
  問題: ブラウザには、Webページの裏側を覗ける道具が内蔵されています。何と呼ばれますか？
  正解: A. 開発者ツール
  解説: 正解は開発者ツールです。ブラウザに最初から入っている「設計図を覗くメガネ」のようなもので、HTML・CSS・JavaScriptの中身を確認できます。
  気になるWebサイトの作り方を、自分で調べられる道具です。
  Q8（穴埋め）
  問題: 構造・見た目・動きをそれぞれ独立に管理する設計思想を何と呼ぶ？
  正解: 関心の分離
  解説: 正解は「関心の分離」です。構造はHTML、見た目はCSS、動きはJavaScript——それぞれの担当を独立させる設計思想です。
  Web技術だけでなく、あらゆるものづくりに通じる普遍的な知恵です。
  Q9（並び替え）
  問題: ソースコードからWebページが表示されるまでの流れを、正しい順番に並べてください。
  正解: サーバー送信 → ブラウザ受信 → HTML解析 → CSS適用 → JavaScript実行 → 画面表示
  解説: サーバーからソースコードが届き、ブラウザがHTML→CSS→JavaScriptの順で画面を組み立てます。
  この仕組みを知っていれば、表示がおかしいときに原因を推測できます。
  Q10
  問題: 設計図の全体像が掴めました。次に深く学ぶべきは、3つの技術のうちどれでしょう？
  正解: A. HTML（骨組み）
  解説: 次はHTML（骨組み）を深く学びます。CSSやJavaScriptから入りたくなりますが、骨組みがなければ壁紙も家電も設置できません。
  HTMLの構造を理解すれば、Webページを「読める人」から「作れる人」への第一歩です。
