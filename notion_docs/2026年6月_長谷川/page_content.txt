1. GitHub Organization 移行の進捗と課題
5月に着手した「GitHub一本化＋PRベース決裁」を、まずCTO室メンバーから先行展開しようとした。
想定よりハードルが高かった。
  個人アカウントにコミットしてしまっている（会社Orgに資産が乗らない）。
  「コミットしているつもり」で実際にはできていない（commit / push 漏れ）。
当初は「このルールでやってね」で回る程度の難易度だと見込んでいたが、実態はそうではなかった。
→ 手順共有だけでは浸透しない。セットアップの型化／伴走が必要だと判明。

3. 7月以降のAI教育戦略
with AIドリル（6月までのコホート型・部署対抗）はいったん一区切り。一方で ドリル自体が現場に良く浸透してきた。
この浸透を土台に、「ドリル → 実践」を繰り返すモデル へ移行する。

4. 7月以降は「実戦編」へ
ドリル ⇄ 実践ループ（7月以降のコアモデル）
ドリル（スマホ・毎日・インプット）に、月次課題に直結する内容を仕込む。
学んだらすぐ PC で実践 → 提出で課題完了、という1サイクルを繰り返す。
現状の AI driven school（座学・ひな形）のボリュームは圧縮し、この「ドリル ⇄ 実践」の往復に主軸を移す。
論点・懸念
スマホ（ドリル）↔ PC（実践）の行き来が若干ややこしい → 導線設計が要検討（ドリル内から「PCでこれを作る」への受け渡し／提出フローをどう滑らかにするか）。

5. データドリブン基盤の構築：全社データのAPI化
各部署のデータ整備が進み、データドリブンで動ける状態に近づいてきた。
これらをすべてAPI化し、役員・幹部がいつでもアクセスできるようにする。
  Claude Code 等のAIが、ほぼリアルタイムの会社情報を把握した状態で動ける ようになり案・判断・実装の精度が上がる。
「人が探して情報を与える」から「AIが常に最新の会社を理解している」を理想とする。
① 組織・部署・スタッフ情報
スタッフ情報・部署情報をAIで構造化。Notion / Discord のチャットを毎日分析し、以下を自動で蓄積・呼び出せるようにする。
  どの部署で・どんな仕事をしているか
  誰との関わりが多いか（コラボ関係・コミュニケーション量）
  いつ入社した人か（在籍期間）
  どんなタイプか（強み・志向・働き方の傾向）
など
「この案件は誰に振るべき？」「この人はどんな人？」「最近この人どう？」にAIが即答できる状態をつくる。
② 集客導線分析（マーケファネル）
勉強会申し込み → 成約 までのデータを一元化。
流入経路ごと・属性ごと に分解し、AIで分析できるようにする。
  流入チャネル別のCVR（申込→成約の歩留まり）
  属性（年齢／職種／きっかけ 等）別のコンバージョン
  どの段階で離脱しているか
→ 勝ち筋の流入・刺さる属性を定量で特定し、集客投資の配分を最適化。
③ 講師データベース
講師まわりの情報を1つのシートに統合（DBとして扱うため、見やすさは度外視）。
  講師アカウント／講師自身のプロフィール
  ローンチ状況・ローンチ成果
  クラス受け持ち状況・クラス情報
  担当生徒の成果
  返信速度、対応漏れなど
完成後は講師ごとの分析が可能に（成果の高い講師の共通項、伸び悩みの要因 など）。
④ 生徒データ分析（入会〜卒業のジャーニー）
入会してから卒業するまでの細かいデータを分析。
講師情報・アイレポートとも紐づけ 、生徒一人ひとりを多面的に把握する。
  学習進捗・到達度
  担当講師との関係性
  成果・つまずきポイント
生徒の成功／離脱の予兆を早期に検知し、フォローにつなげる。
